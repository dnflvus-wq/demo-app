### 3.5 주요 발견: Detail 커버리지가 List보다 낮은 이유

Detail 페이지에서 누락된 4개(DETAIL-001,002,003,009)의 공통점:

- **DETAIL-009 (없는 글)**: 예외 상황 TC로, JS 로직 분석이 필요해 파서로는 감지 불가

이는 Todo 실험에서도 관찰된 패턴: AI는 **인터랙션 TC**(클릭, 입력)를 잘 생성하지만, **데이터 표시 검증 TC**는 약함.

## 5. B-1 vs B-4 비교: 프롬프트 상세도 효과

### 5.1 핵심 비교

| 지표 | B-1 (상세 프롬프트) | B-4 (최소 프롬프트) | 차이 |
|------|:------------------:|:------------------:|:----:|
| 커버리지 | 56.3% (9/16) | **62.5% (10/16)** | **B-4 +6.2%p** |
| 환각 | 15.0% (3/20) | **5.0% (1/20)** | **B-4 -10.0%p** |
| data-testid | 75% | 80% | B-4 +5%p |
| qltyIndex | 4.30 | 4.25 | B-1 +0.05 |
| 생성 시간 | 42.2s | 51.3s | B-4 +9.1s |

### 5.2 놀라운 결과: 최소 프롬프트가 더 나은 이유

**B-4(최소)가 B-1(상세)보다 커버리지 높고 환각 낮음.** 이는 직관에 반하는 결과.

#### 원인 분석:

**1. 상세 프롬프트의 "주의력 편향" (Attention Bias)**
- B-1의 상세 프롬프트는 유효성 규칙(에러 메시지 텍스트 등)을 구체적으로 명시
- AI가 유효성 TC에 과도하게 집중 → 수정 모드(WRITE-005) 등 다른 시나리오 누락
- B-4는 프롬프트 제약 없이 URL에서 추출한 UI spec을 자유롭게 해석 → 수정 모드 자연스럽게 생성

**2. 환각 유발 메커니즘**
- B-1 상세 프롬프트에 "목록에서 자물쇠 아이콘", "카테고리별 뱃지 색상" 등 목록 관련 정보 포함
- AI가 Write 페이지 맥락에서 List 페이지 기능(검색, 빈 목록)을 혼동하여 환각 생성
- B-4는 "게시판 글 작성 페이지의 기능 테스트"만 주었으므로 Write 범위를 벗어나지 않음

**3. URL 파싱의 자체 정보력**
- 개선된 URL 파싱이 form 필드, data-testid, maxlength, select options 등을 정확하게 추출
- 이 정보만으로도 AI가 유효성 TC, 경계값 TC를 충분히 생성 가능
- 프롬프트의 추가 정보가 UI spec과 중복/충돌하면 오히려 품질 저하

### 5.3 B-1만 커버하는 TC vs B-4만 커버하는 TC

| B-1만 매칭 | B-4만 매칭 |
|-----------|-----------|

### 5.4 시사점

1. **상세 프롬프트 ≠ 높은 품질**: 과도한 정보는 AI의 주의력을 분산시킴
2. **URL 파싱이 프롬프트를 상당 부분 대체**: data-testid + maxlength + options 정보가 핵심
3. **프롬프트는 "URL에서 추출 불가능한 정보"만 제공해야 최적**:
   - JS 유효성 규칙 (content 10자 최소 등)
   - 비즈니스 로직 (수정 시 토스트 메시지 등)
   - 없는 기능 명시 (서버 에러 없음 등)
4. **다중 실행 후 합산** 전략이 단일 실행보다 효과적

## 11. B-3 결과 (3 URL 통합) - 서버 타임아웃 실패

### 11.1 실패 상세

| 시도 | 타임아웃 | 결과 |
|------|---------|------|
| 1차 (heredoc 프롬프트) | 300s | TimeoutSec 초과 |
| 2차 (단일라인 프롬프트) | 600s | TimeoutSec 초과 |

3개 URL을 한 번에 제공하면 GENQ 백엔드가 600초(10분) 내에 응답하지 못함.

### 11.2 원인 분석

**비교 데이터:**
- B-1 (1 URL): 42.2초 완료
- B-2 (2 URL): 45.5초 완료
- B-3 (3 URL): 600초+ 타임아웃

2 URL → 3 URL에서 비선형적으로 시간 증가. 가능한 원인:

1. **Playwright 리소스 경합**: 3개 URL을 순차 렌더링 시 브라우저 인스턴스 관리 문제
2. **AI 컨텍스트 폭발**: 3개 페이지의 UI spec이 모두 포함되면 토큰 수가 급증하여 AI API 타임아웃
3. **메모리 압박**: 3개 DOM 트리를 동시에 처리하면서 백엔드 GC 부하
4. **내부 타임아웃 미설정**: 백엔드 Playwright 렌더링에 per-URL 타임아웃이 없어 한 URL이 무한 대기

### 11.3 시사점

- **multi-URL은 2개까지가 안전 한계** (현재 서버 구성 기준)
- Board 전체 TC를 원하면 **B-1 + B-2 분리 실행**이 필수
- GENQ 서버 개선 필요: per-URL 타임아웃 설정, Playwright 인스턴스 풀링
- **이 한계는 GENQ 제품 이슈로 별도 TASK 필요** (서버 Playwright 타임아웃 처리)

### 11.4 B-1 + B-2 분리 vs B-3 통합 비교 (수정된 결론)

B-3가 실패했으므로 분리 전략만 가능:
- **분리 실행 (B-1 + B-2)**: 40 TC 생성, 총 87.7초, Board 전체 커버리지 ~65%
- **통합 실행 (B-3)**: 불가능 (타임아웃)

**결론: 현재 GENQ 서버에서 multi-page TC 생성은 페이지별 분리 실행이 필수.**
